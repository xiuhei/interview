# 常见问题排查

本文档用于快速定位接口报错、日志查看方式、`409 Conflict` 的业务含义，以及前端本地启动时 `vite` 缺失的处理办法。

## 常见状态码

### 400

通常表示请求参数不合法，或者运行前置条件不满足。

常见原因：

- 语音面试提交了文本答案
- 缺少 `audio_file_id`
- 音频文件不存在
- LLM、Embedding、Speech 等环境变量未配置
- 当前会话还没有题目，却提前调用了答题或完成接口

### 401

通常表示登录态无效。

常见原因：

- 没带 `Bearer Token`
- Token 过期或签名不正确
- Token 对应用户不存在

### 403

通常表示权限不够。

常见原因：

- 普通用户访问了管理员接口

### 404

通常表示资源不存在。

常见原因：

- `session_id` 不存在
- `question_id` 不属于当前面试
- 岗位不存在
- 简历不存在
- 音频文件不存在

### 409

通常表示数据库写入冲突，不一定是参数格式错误，而是“这条数据按业务规则不允许重复写入”。

本项目里最常见的含义：

- 同一个 `question_id` 被重复提交答案
- 注册时用户名重复
- 注册时邮箱重复

### 500

通常表示服务内部异常。

常见原因：

- 代码异常未被业务逻辑捕获
- 数据结构和代码假设不一致
- 第三方接口返回了未处理的错误

## 怎么看日志

日志目录：

- [var/logs/access.log](/c:/a/code/interview/var/logs/access.log)
- [var/logs/app.log](/c:/a/code/interview/var/logs/app.log)
- [var/logs/error.log](/c:/a/code/interview/var/logs/error.log)

建议排查顺序：

1. 先看 `access.log`，确认报错接口、时间、状态码和 `request_id`
2. 再用同一个 `request_id` 去 `app.log` 或 `error.log` 里搜详细堆栈
3. 如果是数据库问题，重点搜 `IntegrityError`、`Duplicate entry`、`ForeignKey`、`NOT NULL`

PowerShell 常用命令：

```powershell
Get-Content data\logs\access.log -Tail 100
Get-Content data\logs\app.log -Tail 100
Get-Content data\logs\error.log -Tail 100
```

按 `request_id` 搜索：

```powershell
Select-String -Path data\logs\*.log -Pattern "984ac079463c478288e7813a34b7f1ec"
```

按状态码搜索：

```powershell
Select-String -Path data\logs\access.log -Pattern "status=409"
```

按数据库冲突搜索：

```powershell
Select-String -Path data\logs\app.log,data\logs\error.log -Pattern "IntegrityError|Duplicate entry"
```

## 为什么会返回 409

当前后端在捕获数据库 `IntegrityError` 时，会统一转换成 `409 Conflict`。

参考代码：

- [backend/app/core/exceptions.py](/c:/a/code/interview/backend/app/core/exceptions.py#L41)

这意味着只要数据库写入违反了唯一约束、外键约束或其他完整性规则，接口层就会返回 `409`。

### 这次 409 的实际原因

从日志可以确认，这次不是模型输出问题，也不是评分逻辑问题，而是数据库唯一键冲突：

- `POST /api/v1/interviews/19/answers status=409`
- `Duplicate entry '48' for key 'interview_answers.uq_interview_answers_question_id'`

这说明：

- `question_id=48` 已经存在一条答案
- 后端又尝试为同一题再插入一条答案
- 因为 `interview_answers.question_id` 是唯一的，所以数据库拒绝写入

约束定义在：

- [backend/app/models/interview_answer.py](/c:/a/code/interview/backend/app/models/interview_answer.py#L11)

答案写入点在：

- [backend/app/services/interview_service.py](/c:/a/code/interview/backend/app/services/interview_service.py#L202)

### 看到 409 后怎么判断

如果日志里出现下面这种信息：

```text
Duplicate entry '48' for key 'interview_answers.uq_interview_answers_question_id'
```

可以直接判断为：

- 同一题被重复提交答案

如果以后看到别的唯一键名，就按键名判断冲突类别：

- 用户名唯一键：用户名重复
- 邮箱唯一键：邮箱重复
- `question_id` 唯一键：同一题重复答题

## 前端 `vite` 找不到怎么办

如果你在 `frontend` 目录运行：

```powershell
npm run dev
```

看到类似报错：

```text
'vite' 不是内部或外部命令，也不是可运行的程序或批处理文件
```

这通常不是 `package.json` 缺脚本，而是前端依赖安装已经损坏。

### 怎么确认是不是依赖损坏

项目正常时，下面这些文件应该存在：

- `frontend/node_modules/.bin/vite.cmd`
- `frontend/node_modules/vite/package.json`

如果 `node_modules` 目录还在，但上面两个文件不存在，说明 `node_modules` 是残缺状态。

也可以运行：

```powershell
npm --prefix frontend ls vite
```

如果出现 `invalid`，也说明当前依赖不完整。

### 修复办法

在项目根目录执行：

```powershell
Remove-Item frontend\node_modules -Recurse -Force
Remove-Item frontend\.npm-cache -Recurse -Force
npm --prefix frontend ci
```

安装完成后再验证：

```powershell
npm --prefix frontend ls vite
Test-Path frontend\node_modules\.bin\vite.cmd
```

预期结果：

- `npm ls vite` 不再报 `invalid`
- `Test-Path` 返回 `True`

然后再启动：

```powershell
cd frontend
npm run dev
```

### 脚本化修复

如果你走本地初始化脚本：

```powershell
.\scripts\bootstrap_local.ps1
```

脚本现在会：

- 清理损坏的 `frontend/node_modules`
- 清理 `frontend/.npm-cache`
- 执行 `npm ci`
- 额外验证 `vite` 可执行文件和依赖树是否完整

这样可以更早发现“依赖目录存在，但其实已经坏了”的情况。

## 一句话结论

- `409` 的本质是数据库完整性冲突，这次是同一题重复提交答案
- 前端 `vite` 找不到的本质是依赖安装损坏，不是脚本缺失
